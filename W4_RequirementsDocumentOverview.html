Requirements Document Review

Requirements Document Outline

  • Executive Summary
  • Business Objectives
  • Business Requirements
  • Technical Requirements
  • Operational And Performance Requirements
  • User Acceptance Testing Requirements
  • Implementation Requirements

Executive Summary Section

  • Provide an overview of the project and what needs your software will address
    • System Overview – describe the solution and its components
    • System Objectives – describe the business problem the system will solve
    • Key System Functions – Identify high-level functionality for the system
      • Use information from the user stories to complete this section

Business Objectives Section

  • Describe the business, commercial, or academic need for your solution
    • Overview – identify high-level business objectives (e.g., “Sell more sneakers”, “Increase University Enrollment”)
    • Corporate Goals – ignore this section
    • Statement of Business Objects – ignore this section
    • Current Process – ignore this section (including context diagram)
    • Proposed Solution – describe how the solution will be used by employees, customers, or other stakeholders
      • Include a Use Case, or other diagram, to identify how different users will interact with different components of your solution

Business Requirements Section

  • Provide detailed requirements for your system, using User Stories and Product Backlog Items to develop requirements in the MoSCoW format
    • Business Functions – describe the functions of your solution (e.g., track inventory, scan barcodes, track walking activity, conduct online surveys, provide online chat responses to students)
    • Functional Requirement- using the MoSCoW format, create the requirements necessary to enable the business functions in your solution
    • Maintenance Functions – ignore this section
    • Report Requirements – using the MoSCoW format, create the reporting requirements for your solution

Technical Requirements Section

  • Describe the non-functional requirements for your solution
    • Hardware and System Software requirements – identify the hardware (e.g., computer, phone, tablet, server) and software (e.g., Android vX.X, iOSvY) required to run your solution
    • Software System Interface requirements – identify the other software (if any) that your solution requires to run (e.g., access to Google Maps via API on device, interface to barcode scanner firmware)
    • Commercial-off-the-Shelf (COTS) software – identify (if any) COTS software components that you plan to integrate into your solution

Operational and Performance Requirements Section

  • Describe the non-functional requirements for your solution
    • Operation mode and Availability – identify the operation mode (e.g., real-time, interactive) and availability requirements for users
    • Availability – identify the up-time and other availability requirements for your solution
    • Capacity requirements – identify the number of concurrent users, concurrent transactions, and other capacity requirements for your solution
    • Storage requirements – identify how much storage is required for your solution and where that storage must be (e.g., on a device, on a server)
    • Facility, Co-location, Infrastructure or Co-hosting requirements – ignore section
    • Network or Connectivity requirements – identify the bandwidth (network, wireless, mobile data) requirements for your solution
    • Security requirements – identify requirements for handling authentication and protecting data in your solution
    • Post-Implementation Support Concepts – ignore section

User Acceptance Testing Requirements Section

  • Ignore this section

Implementation Requirements Section

  • Describe how your solution will be rolled-out to users
    • Training requirements – describe the users to be trained and how much training will be required
    • System roll-out requirements – describe constraints and schedule for rolling-out your solution
    • Communication requirements – describe how communications to stakeholders will be done (e.g., status updates will be sent to stakeholders every 2 weeks)
    • Operational procedures – ignore section
    • User Documentation requirements – identify the user documentation required for operation of the solution (e.g., user’s guide, quick reference guide, online help)
    • System Distribution, Facility Readiness and Built-in Help requirements – ignore these sections
